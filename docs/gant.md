# Гант нового поколения: real-time detailed progress bar (концепт)

> Концепт-документ по будущему движку визуализации покрытия данными. Реализация — [phase 12](./dev/phase12/plan.md)
> (`FUTURE`). Текущий MVP (DOM-колбаски) остаётся в проде, пока не станет узким местом.

## 1. Что это на самом деле

То, что мы называем «Гант», по факту **не диаграмма Ганта**. Это:

> **Real-time detailed progress bar** на каждый инструмент — плотный, кластеризованный по времени
> прогресс-бар присутствия/интенсивности данных, помещённый в **зумируемую и перетаскиваемую сцену**
> (zoom & drag) с **LOD** (level-of-detail) поверх предагрегатов TimescaleDB.

Модель одной «колбаски»:

- Ось времени бьётся на **бакеты**; базовое разрешение — **~30с ≈ 1px** (при D1).
- В каждый бакет (1px) закладывается **агрегация**: были ли сделки и сколько (**интенсивность** →
  насыщенность/цвет). То есть колбаска — это **набор 1px-полосок**, а не «интервал со скруглением».
- Цвет полоски = источник данных (мультиисточник, `source_id`); интенсивность = плотность сделок.

Ключевой сдвиг мышления: **колбаска — это растр (изображение по бакетам), а не набор элементов.**
Отрисовка тысяч прямоугольников и плавный zoom/pan — задача GPU, а не верстки.

## 2. Почему текущий подход (DOM) упрётся

Сейчас `CoverageTrack` (`web/src/ui/components/CoverageTrack.tsx`) рисует **DOM-элементы**
(`div`/`span`) на каждый сегмент покрытия, gap и зону сессии. Виртуализация строк
(`useVirtualRows`) ограничивает вертикаль (рисуем ~20–30 видимых инструментов), но целевая
бакетная модель по горизонтали даёт **сотни–тысячи полосок на строку**:

```
ширина трека (px) × видимые строки × перерисовка по realtime-тику
≈ 1500 бакетов × 30 строк × 1 Гц роста = десятки тысяч узлов, живущих и мутирующих
```

DOM здесь — неправильный инструмент: платим за layout / reflow / style recalc / composite там, где
нужно просто «залить прямоугольники». При «сотнях инструментов на записи» lag-ов не избежать. Это не
оптимизируется косметикой — упирается в саму модель «элемент на полоску».

## 3. Почему **не canvas 2D** (как целевой)

Canvas 2D — шаг лучше DOM, но не цель:

- Сцена **пересобирается каждый кадр на main-thread**. Zoom = пересчитать позиции всех видимых
  прямоугольников и заново залить их, **каждый кадр анимации**. Это CPU-bound.
- Плавный инерционный zoom/pan над десятками тысяч примитивов начинает ронять кадры — ровно наш
  сценарий (растягивание колбасок + горизонтальный скролл в реальном времени).
- Грубый потолок: canvas 2D комфортно тянет ~10–30к `fillRect`/кадр в статике; при непрерывном зуме
  с пересборкой сцены — существенно меньше.

Вывод: **canvas 2D оправдан только как краткосрочный stopgap** над DOM (если нужен быстрый выигрыш
«на неделю»), но не как движок под real-time zoom/pan. Если аппетит на нормальный инструмент —
canvas можно пропустить.

## 4. Почему **не сырой WebGL**

WebGL2 — правильный целевой слой (GPU-bound): геометрия заливается в буферы **один раз**, а zoom/pan
= смена **uniform** (матрица проекции / вьюпорт); GPU переспроецирует сотни тысяч–миллионы примитивов
за кадр без участия CPU. Это каноничный кейс WebGL.

Но **писать на сыром WebGL — утонуть**: ручное управление буферами/атрибутами/шейдерами, VAO,
батчинг, ресайз, DPI, потеря контекста, текстовые лейблы, hit-testing. Недели инфраструктуры до
первого пикселя. Берём тонкую библиотеку.

## 5. Выбор фреймворка

Сравнение кандидатов под нашу задачу (масштаб — много **простых прямоугольников** с instancing,
плавный zoom/pan, realtime-обновление, интеграция в React/RxJS):

| Кандидат | Что это | Плюсы под нас | Минусы / риск |
| -------- | ------- | ------------- | ------------- |
| **regl** | Функциональная обёртка над WebGL(2), декларативные draw-команды | Минимализм, идеален для **instanced-прямоугольников**; полный контроль над шейдером (важно для проекции оси); крошечный | Нет сцен-графа/текста/интерактива — пишем сами (для нас это ок: сцена простая) |
| **PixiJS** | 2D WebGL/WebGPU рендер-движок (спрайты, графика, текст, события) | Сцен-граф, текст, интерактив, ресайз из коробки; активное развитие; v8 умеет WebGPU | Заточен под спрайты/UI; массовые тонкие прямоугольники — через `Graphics`/mesh, надо аккуратно батчить; больше «магии» |
| **deck.gl** (luma.gl) | GPU-визуализация больших данных (слои, viewport, transitions) | Из коробки: viewport, zoom/pan, LOD-мышление, огромные датасеты; слой `SolidPolygon`/кастом | Геозаточен (хотя есть `OrthographicView`); тяжелее; кастомный слой под «bucket-бар» = свой шейдер всё равно |
| **uPlot** | Сверхбыстрый **canvas 2D** чартинг | Очень быстрый для 2D-графиков, крошечный | Canvas 2D (см. §3) — не решает zoom-потолок; заточен под линии/оси, не под наш растр |
| **Konva** | Canvas 2D сцен-граф | Удобный интерактив | Canvas 2D — тот же потолок |
| **Perspective (FINOS)** | WebAssembly-движок таблиц/графиков | Мощно для аналитики | Избыточно, своя модель данных |
| сырой **WebGL2** | — | Максимальный контроль | Дорого по времени (§4) |
| **WebGPU** | Новый GPU-API | Будущее, лучший compute | Зрелость/поддержка browsers — риск на сегодня; берём как «потом» |

**Рекомендация:** начать с **regl** — наша сцена по сути «инстансированные прямоугольники + кусочная
проекция времени в vertex-шейдере», а regl даёт ровно контроль над шейдером при минимуме обвязки.
**PixiJS** — резервный вариант, если понадобится богатый сцен-граф/текст/события «из коробки».
**deck.gl** — если пойдём в сторону «одна большая сцена со всеми строками + встроенный viewport/LOD»
и готовы принять его модель. **WebGPU** — целевой горизонт, но не первый заход.

## 6. Что уже есть и играет на руку

Ряд принятых решений идеально ложится на GPU-рендер — миграция не потеряет функциональность:

- **Проекция оси вынесена в чистые функции** — `web/src/core/sessionProjection.ts`
  (`makeProjector` / `makeInverseProjector`). Маппинг `ms → x` и обратно уже отделён от рендера;
  его можно перенести в vertex-шейдер или переиспользовать для hit-testing.
- **Интерактив по времени — уже математический, не DOM.** Вертикальный crosshair
  (`web/src/core/crosshair.ts` + `CrosshairOverlay`) и позиция курсора считаются через inverse
  projector. Нативные `title`/hover с колбасок убраны → при переезде на canvas/WebGL ничего не теряем.
- **Данные уже агрегированы на бэке** — `coverage_segment` (phase 6a) даёт интервалы+разрывы; это
  отправная точка для перехода к бакетам/интенсивности.
- **Стор фреймворк-агностичен** (RxJS `sessions$` / `window$` / `coverage$` в `OhsStore`) — рендерер
  подписывается на потоки, React не в горячем пути отрисовки.
- **Виртуализация строк** уже есть — GPU-слою нужно рисовать только видимый диапазон.
- **API `CoverageTrack`** (`window` / `segments` / `sessions`) стабилен — целимся в **drop-in замену**
  компонента, не переписывая вызывающий код.

## 7. Подводный камень: zoom ⟂ посессионная проекция

Главная концептуальная развилка. Непрерывный zoom подразумевает **линейную ось времени**, а текущая
ось **нелинейна**: сессии схлопнуты, неторговые разрывы — в шов, выходные короче (см. `promt.md §4`,
`sessionProjection.ts`). Это конфликт: линейная матрица зума не совместима с кусочной проекцией.

Варианты (решить **до** кода):

1. **Кусочно-линейная проекция в vertex-шейдере** — маппинг `ms → x` по сессиям выполняется на GPU
   (передаём границы сессий как uniform/texture). Сохраняет схлопнутую модель при любом зуме, но
   шейдер сложнее, и «плавность» зума на стыках сессий требует аккуратности.
2. **Два режима сцены:**
   - **«Обзор»** — посессионная схлопнутая ось (как сейчас): много инструментов, дискретные D/W/M.
   - **«Анализ / зум»** — линейное реальное время, плавный zoom/pan/inertia, LOD по глубине.

   Вероятно **честнее для UX**: при глубоком зуме схлопывание всё равно теряет смысл (хотим видеть
   реальные интервалы и тишину). Режимы переключаемы; «Обзор» может остаться даже на текущем DOM/canvas.

Рекомендация — **режимы** (вариант 2) как основной, с возможностью позже занести кусочную проекцию в
шейдер для гибрида.

## 8. LOD + Timescale continuous aggregates (фундамент)

Потолок FPS упрётся **в данные**, не в рендерер. Даже GPU нельзя кормить «все сделки всех
инструментов». Нужен **LOD**: не рисуем больше бакетов, чем пикселей; при зуме подгружаем более
мелкое разрешение — как **тайлы в картах**.

- **TimescaleDB continuous aggregates** — предматериализованные роллапы на нескольких разрешениях
  (напр. 30с / 5м / 1ч / 1д): `time_bucket(...)` + `count(*)`/`bool_or(...)` → покрытие + интенсивность.
  Обновляются инкрементально (realtime-политика Timescale), поэтому «хвост» бара свежий без пересчёта
  всей истории.
- **API отдаёт бакеты под текущий zoom** (разрешение выбирается по `пиксели/окно`), формат —
  **компактные типизированные массивы** (напр. 1 байт интенсивности на бакет на источник), не
  JSON-объекты на сегмент.
- **Важно:** эта работа **ортогональна выбору рендерера** и не пропадает — фундамент нужен и для
  canvas, и для WebGL, и для WebGPU.

## 9. Прагматика (когда и как мигрировать)

- **Сейчас MVP (DOM) доживёт** — пока строк на экране немного и сегментов мало. Не переписываем
  преждевременно.
- **Триггер миграции:** первый заметный jank при «сотнях инструментов на записи» + realtime-росте.
- **Миграция локальна:** заменяем `CoverageTrack` на GPU-реализацию с тем же props-API
  (`window/segments/sessions/highlightDays`), остальной `InstrumentPicker` не трогаем.
- **Порядок работ (черновой A→B→C, см. phase 12):**
  - **A. Рендерер** — regl-слой `CoverageTrack` (instancing, только видимые строки, рост live-края
    без полной пересборки).
  - **B. LOD-агрегация** — Timescale continuous aggregates + API бакетов под zoom.
  - **C. Zoom/pan** — инерционный горизонтальный zoom/pan; выбранный режим оси (§7); при
    необходимости `OffscreenCanvas` + Web Worker (агрегация/декодирование вне main-thread).

## 10. Выбор стека (итог)

| Слой | Выбор | Обоснование |
| ---- | ----- | ----------- |
| Рендерер (цель) | **WebGL2 через `regl`** | GPU zoom/pan через uniform, минимум обвязки, контроль шейдера под проекцию; PixiJS — резерв, deck.gl — при «единой сцене», сырой WebGL — нет |
| Рендерер (stopgap, опц.) | Canvas 2D | Только если нужен быстрый выигрыш над DOM до WebGL |
| Ось при зуме | **Два режима** (Обзор / Анализ-зум) | Схлопнутая ось несовместима с плавным зумом; кусочная проекция в шейдере — как будущий гибрид |
| Данные / LOD | **TimescaleDB continuous aggregates** | Multi-resolution бакеты (покрытие+интенсивность), инкрементальное обновление; компактные типизированные массивы по API |
| Вне main-thread (при масштабе) | `OffscreenCanvas` + Web Worker | Агрегация/декодирование не блокируют UI |
| Горизонт | **WebGPU** | Целевой «потом», когда дозреет поддержка |
