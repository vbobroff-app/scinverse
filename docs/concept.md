# Scinverse — концептуальные решения

> Документ фиксирует принятые архитектурные решения и их обоснование.
> Связанные документы: [`ohs.md`](./ohs.md) — Online History Server.

---

## Клиентская стратегия / визуализация

### Решение

**Один web-клиент (React + WebGL/WebGPU) как единственный UI.** Отдельного «боевого» десктоп-приложения не делаем. Нативный десктоп сознательно отложен и допускается только как **тонкий потребитель того же API** (не как параллельный стек).

### Обоснование

Исторически трейдинг-софт (WealthLab, TSLab, BackTrader, QScalp) был десктопным, но объём данных — не главная причина. Настоящие причины десктопа в архитектуре Scinverse **уже сняты, потому что движок вынесен на сервер**:

| Причина «десктопности» софта | Как закрыто в Scinverse |
| :--- | :--- |
| Коннекторы брокеров — нативные Windows-DLL (Plaza2, QUIK, Transaq) | Коннекторы живут в C#-сервере (OHS/Агент); клиент их не касается |
| Латентность ручного клика «цена → ордер» | Торгует автоматический C#-агент в горячем контуре; UI не на критическом пути |
| Локальный тяжёлый компьют (бэктест) | Исследования — на Python (отдельный контур) |

Раньше «приложение = движок». В Scinverse движок (данные, логика, исполнение) уже на сервере, клиенту остаётся **визуализация и управление** — задача, терпимая к латентности, для которой web подходит идеально.

Современный web (WebGL2/WebGPU, WebSocket, WASM, Web Workers, OffscreenCanvas) свободно тянет 100k+ свечей на 60fps и живой стакан. Ограничения 2000-х (не было WebGL/WebSocket/WASM) сегодня неактуальны.

### Почему не монолит «коннектор + ТС» ради латентности

- **Для автостратегий** латентность уже оптимальна: решение принимает агент в одном процессе с коннектором. Клиент не на критическом пути.
- **Для ручного клика** монолит экономит один hop. У тонкого клиента на localhost это loopback-WebSocket (<1 мс). Дальнейший путь `клиент → C# OMS → брокер → биржа` одинаков для любого клиента и доминируется сетью до биржи. На фоне реакции человека (~150 мс) loopback-хоп неразличим.

Вывод: выигрыш монолита — микросекунды, тонущие в сетевом пути; цена — второй стек и вечно отстающий legacy. **Не оправдано.**

### Режимы развёртывания одного web-бандла

Одна кодовая база — три режима без смены технологии:

1. **Удалённо в браузере** — обычный сценарий, доступ откуда угодно.
2. **Локально на хосте сервера** — тот же бандл по loopback → near-desktop латентность для ручного DOM.
3. **В лёгкой desktop-оболочке** (Tauri / WebView2) — «вид приложения», трей, хоткеи, окно поверх. По-прежнему web-технологии и один код.

Оболочка (если появится) остаётся **чистым потребителем Query/Live API**. Нельзя позволять ей отрастить нативные «щупальца» в коннектор — иначе возврат к монолиту и legacy-дрейфу.

### Когда десктоп всё-таки оправдан (не сейчас)

Единственный сценарий — **ручной скальпинг-кокпит** в стиле QScalp (клик по стакану с минимальной задержкой). Это ниша, противоречащая ставке на автоматических агентов. Если понадобится — реализуется как тонкий клиент над тем же API (см. режим 2/3), а не отдельным стеком.

### Технические заметки (footprint на web)

- Тела/кластеры footprint — легко на WebGL/WebGPU (тысячи прямоугольников).
- Текст в кластерах (объёмы на ценовых уровнях) — самое дорогое на GPU. Приём: гибрид — тела на WebGL, текст на `Canvas2D` слоем сверху, с LOD (не рисовать цифры мельче N px) и интервальным троттлингом.
- Троттлинг живого потока — **на сервере** (агрегация + дельты по WebSocket, раз в 5–15 мс), а не сырой firehose в UI. Это верно для любого клиента, поэтому десктоп тут не даёт преимущества.

### Что берём из legacy

Из старых наработок (`C:\sources\legacy\...`) переносим **модели и уроки, а не код и не стек** (.NET 4.5 + WPF/DrawingVisual — Windows-only, экосистема застыла):

- доменные модели `IFootprint` / `IClaster` (бар + словарь кластеров по цене с buy/sell);
- модель стакана (`IOrderSouce`: `BuyQue`/`SellQue` как `SortedDictionary<Int64, IOrder>`, `ISpread`);
- принцип цены в шагах (`NormPrice` → `price_ticks`, уже заложен в `ohs.md`);
- уроки рендеринга: LOD (`MinFontSizeVisible`), интервальный троттлинг перерисовки.

Ключевое условие при портировании — **развести домен и отрисовку**: footprint/кластеры/стакан должны быть чистыми DTO (без WPF-типов), чтобы их одинаково потребляли и web-клиент, и возможная desktop-оболочка.

### Наблюдаемость сервера ≠ график

Отдельный «серверный» график не нужен. Здоровье OHS (идёт ли сбор, гэпы, лаг записи) наблюдаем через **Grafana + health-API**, а не через чарт. График и DOM — пользовательская фича web-клиента.
