# Phase 12. Гант-рендер: переход MVP → настоящий графический движок (WebGL2 + LOD)

Крупная веха: замена MVP-отрисовки покрытия (DOM `div`/`span` на инструмент) на масштабируемый
графический слой с **real-time zoom/pan** по горизонтали над сотнями–тысячами инструментов.

> Концепт и обоснование выбора стека (почему не DOM/canvas/сырой WebGL, сравнение regl/PixiJS/deck.gl,
> подводный камень zoom ⟂ проекция) — в [`docs/gant.md`](../../gant.md). Здесь — план реализации.

**Статус:** `FUTURE` (регистрируем решение; стартует после того, как «быстрая графика» станет узким
местом — при сотнях инструментов на записи). **Stage:** 1 (admin frontend), поздний этап.
**Зависимости:** phase 7b (посессионная проекция/`sessions$`/`window$`), phase 7f (тайм-лайн-фильтр,
inverse projector, crosshair), phase 6a (`coverage_segment`).

## Мотивация

Сейчас каждая колбаска — набор DOM-элементов (сегменты/gap/зоны). Виртуализация спасает по вертикали,
но целевая модель — бакетизированный progress-bar (**~30с ≈ 1px**, интенсивность = плотность сделок в
бакете) — по горизонтали даёт сотни–тысячи полосок на строку. `ширина × строки × реалтайм-тик` =
десятки тысяч узлов → неизбежный lag. Колбаска по сути — **растр**, а не набор элементов; отрисовка
прямоугольников и плавный zoom/pan — задача GPU.

## Ключевое решение: рендерер

- **Не canvas 2D как целевой** — сцена пересобирается каждый кадр на main-thread; непрерывный zoom над
  десятками тысяч примитивов роняет FPS. (Canvas оправдан лишь как краткосрочный stopgap над DOM.)
- **Целевой — WebGL2 через тонкую библиотеку** (`regl` — instancing прямоугольников; либо `PixiJS`).
  Геометрия в GPU-буферах один раз, zoom/pan = смена uniform (матрица/вьюпорт), GPU переспроецирует
  без участия CPU. Сырой WebGL не берём (экономим недели). WebGPU — «когда дозреет», не сейчас.
- DOM остаётся только для обвязки (строки/лейблы/тумблеры). Интерактив по времени уже математический
  (inverse projector + crosshair) — переносится на GPU без потерь.

## Ключевое решение: данные (LOD) — фундамент, ортогональный рендереру

Потолок FPS упрётся в данные, не в рендерер. Нужен **LOD**: не рисуем больше бакетов, чем пикселей;
при зуме подгружаем более мелкое разрешение (как тайлы в картах).

- **TimescaleDB continuous aggregates** — предматериализованные роллапы (30с / 5м / 1ч / …): покрытие
  + интенсивность (`time_bucket` + `count`). Фронт тянет разрешение под текущий zoom.
- Формат — компактные типизированные массивы (напр. 1 байт интенсивности на бакет), не JSON-объекты
  на сегмент.
- Эта работа не пропадает при любом выборе рендерера.

## Открытый вопрос (решить до кода): zoom ⟂ посессионная ось

Непрерывный zoom подразумевает линейную ось времени, а текущая ось нелинейна (сессии схлопнуты,
выходные короче). Варианты:

1. Кусочно-линейная проекция в **vertex shader** (ms→x по сессиям на GPU) — сохраняет модель при зуме.
2. **Два режима**: «Обзор» (посессионная схлопнутая, как сейчас) и «Анализ/зум» (линейное время,
   плавный zoom/pan). Вероятно честнее для UX — при глубоком зуме схлопывание теряет смысл.

## Этапы (черновой A→B→C)

- **A. Рендерер.** WebGL2-слой `CoverageTrack` (drop-in по API `window/segments/sessions`): instancing
  прямоугольников, отрисовка только видимых строк, рост live-края по тику без полной пересборки.
- **B. LOD-агрегация.** Timescale continuous aggregates (multi-resolution), API отдаёт бакеты под zoom,
  клиент запрашивает разрешение по ширине/окну.
- **C. Zoom/pan.** Инерционный горизонтальный zoom/pan; решение по режиму оси (см. выше);
  при необходимости `OffscreenCanvas` + Web Worker для агрегации/декодирования вне main-thread.

## Критерии приёмки (черновые)

1. Плавный (60 fps) горизонтальный zoom/pan при N сотнях видимых инструментов.
2. Интенсивность бакета отражает плотность сделок; LOD переключается по zoom без «дребезга».
3. Реалтайм-рост колбасок без деградации FPS.
4. Функциональный паритет с MVP (цвет=источник, gaps, зоны Full+сессия, crosshair, стандарт времени).

## Вне области

- Полноценный чартинг цены/стакана (это отдельный трек `ui-charting`).
- WebGPU-путь (фиксируем как последующий, когда стабилизируется).
