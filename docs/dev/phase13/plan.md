# Phase 13. Кэширование (сквозное)

Единый слой кэширования для **всей системы**, а не точечные кэши в отдельных сервисах. Обобщает
in-memory-кэш ISS из phase 7c (`IssExchangeCatalog` поверх `IMemoryCache`) в переиспользуемую
инфраструктуру с персистентностью, отказоустойчивостью к флаки-апстримам и внятными политиками
инвалидации.

**Статус:** `PLANNED` (регистрируем решение; стартует независимо, когда рестарты/флаки-апстримы/
нагрузка станут ощутимы). **Stage:** 1, сквозная инфраструктура.
**Зависимости:** phase 7c (первый потребитель — ISS-каталог; снимает его `TODO (7c.3)`).

## Мотивация

- **Флаки-апстримы.** MOEX ISS «отдаёт через раз» — прокси с кэшем уже делает нестабильность
  невидимой, но только пока запись жива (TTL). После протухания сбой апстрима = ошибка пользователю.
- **Рестарты.** `IMemoryCache` живёт в памяти процесса Host: рестарт → кэш пуст → первый запрос снова
  тяжёлый (списки опционов — десятки тысяч строк).
- **Не только ISS.** Кэшировать нужно и read-model'и каталога, расписания/календари, тяжёлые агрегаты
  Ганта и прочие дорогие вычисления. Сейчас каждый потребитель решает это сам (или никак).
- **Масштабирование.** При нескольких инстансах Host in-memory-кэш не шарится → дублирующая нагрузка
  на апстрим и рассинхрон.

## Область (что кэшируем)

- Внешние источники: MOEX ISS (структура биржи, расписания, справочники), позже — другие площадки.
- Внутренние read-model'и: группировки инструментов, значения фильтров каталога.
- Тяжёлые вычисления: агрегаты/роллапы покрытия и активности (пересекается с LOD phase 12).

## Ключевые решения (обсудить до кода)

1. **Абстракция.** Единый интерфейс `ICache`/`ICachedSource<T>` (get-or-fetch, инвалидация, метаданные
   свежести) — потребители не знают о бэкенде. ISS-каталог переводится на него первым.
2. **Бэкенд(ы) кэша.**
   - L1 in-memory (как сейчас) + L2 персистентный.
   - Персистентный: **БД (Postgres/Timescale)** — переживает рестарт, основа для отслеживания
     обновлений по `updatetime`/ETag; либо **Redis** — если нужен общий распределённый кэш между
     инстансами и TTL «из коробки». Выбор — по требованиям к мультиинстансовости.
3. **Stale-on-error.** Хранить последнюю успешную копию без жёсткого удаления по TTL: при протухании
   пробуем апстрим, при сбое/таймауте отдаём «устаревшую, но валидную» с пометкой (`X-Stale`/флаг).
4. **Refresh-ahead.** Фоновое обновление записи незадолго до истечения TTL, чтобы пользователь всегда
   попадал в тёплый кэш (single-flight — не допускать «стада» одновременных обновлений одного ключа).
5. **Политики по видам данных.** Разные TTL/инвалидация: структура биржи (часы), списки инструментов
   (десятки минут), справочники (по кнопке/событию), агрегаты (по подписке на новые данные).
6. **Инвалидация.** По TTL, по `updatetime`/ETag апстрима (авторитетно), по событию (напр. рефреш
   справочника классов), ручная (админ).
7. **Наблюдаемость.** Метрики hit/miss/stale-served/refresh, размеры, возраст записей.

## Этапы (черновой A→B→C)

- **A. Абстракция + L1.** Ввести `ICache`/get-or-fetch, перенести `IssExchangeCatalog` на неё без смены
  поведения (in-memory), закрепить тестами. Параметризовать TTL через `OhsOptions`.
- **B. Персистентность + stale-on-error.** Добавить L2 (БД или Redis), сохранять последнюю валидную
  копию, отдавать stale при сбое апстрима, прокидывать признак «устаревшее» в ответ/UI.
- **C. Refresh-ahead + метрики.** Фоновое обновление (single-flight), метрики hit/miss, при
  необходимости распределённость (общий L2 между инстансами).

## Критерии приёмки (черновые)

1. ISS-каталог работает через общий слой кэша; поведение не хуже текущего (7c), тесты зелёные.
2. После рестарта Host первый запрос популярного борда обслуживается из персистентного кэша (не бьёт
   в ISS повторно, если запись свежа).
3. При недоступности ISS и протухшем TTL пользователь получает последнюю валидную копию с пометкой
   «устаревшее», а не 502.
4. Видны метрики hit/miss/stale-served.

## Вне области

- Кэш статики фронта/CDN (решается на уровне веб-сервера/сборки).
- LOD-агрегация Ганта как таковая (phase 12) — здесь только кэш/материализация её результатов, если
  понадобится пересечение.
