# Phase 7 — MVP → Release: швы, переход, сложности

Где живёт вся **forward-looking** информация о переходе от MVP-админки (фаза 7) к release-качеству:
**швы** (что делает переезд инкрементальным), **известные сложности/проблемы**, и **что осознанно
временно**. Заведён отдельно от [phase 12](../phase12/plan.md) / [`gant.md`](../../gant.md) специально:
те доки — глубокий разбор release-движка «когда MVP устаканится», а план там **сырой и ещё передумается**.
Здесь — рабочие заметки уровня фазы 7, обновляются по ходу прототипирования.

См. также обзорную карту фазы — [`roadmap.md`](roadmap.md).

## Принцип: временное vs долговечное

MVP = «работает, но не правильно/быстро». Чтобы это не превратилось в переписывание на release, каждое
**временное** решение прячем за **стабильным швом**. Тогда release = замена реализации за швом, а не
хирургия по всему коду. Ниже — карта швов и то, что за ними временно.

## Швы, которые уже стоят (переезд будет инкрементальным)

| Шов (стабильный контракт) | Что за ним сейчас (временно) | На что заменится (release) |
| ------------------------- | ---------------------------- | -------------------------- |
| `ITradeActivityStore(ids, source, from, to, bucket)` | app-кэш (`trade_activity_bucket` + `_computed`), ленивый расчёт по дням | **иерархические continuous aggregates** (LOD multi-resolution) |
| Статическая лесенка `bucketSecondsForTimeframe` (~7 ступеней) | фронт выбирает разрешение по числу дней | тайловый LOD «пикселей не больше, чем бакетов» |
| `sessionProjection.ts` (`makeProjector`/`makeInverseProjector`) | маппинг `ms↔x` на CPU (чистые функции) | тот же маппинг в **vertex-шейдере** / для hit-testing |
| Props `CoverageTrack` (`window/segments/sessions/activity…`) | DOM `div`/`span` на бакет | **drop-in WebGL2 (regl)** с тем же API |
| RxJS-сторы (`coverage$`/`window$`/`sessions$`/`activity$`/`liveness$`) | React подписан на сторы | рендерер подписывается на те же потоки, React не в горячем пути |
| `crosshair.ts` + inverse projector | интерактив по времени уже **математический**, не DOM | без потерь переносится на canvas/WebGL |
| Доменные сторы бэка (`ICoverageStore`, `ICaptureLivenessStore`, `IMarketConnector`) | Timescale + TRANSAQ DLL / synthetic | другие источники/хранилища без смены хоста |

Ключевой вывод: **пропасти MVP→release нет** — есть набор швов; release дробится на независимые замены.

## Известные сложности / проблемы перехода

1. **Разрыв разрешения (LOD).** Лесенка дискретна (~7 ступеней) → при смене зума разрешение «скачет».
   *Release:* иерархические continuous aggregates (cagg-на-cagg 30с→5м→1ч→1д) = дешёвый тайловый LOD с
   инкрементальным свежим хвостом. *Сейчас (косметика):* keep-previous-while-loading (не чистить слой до
   прихода новых бакетов) + гистерезис/дебаунс переключения ступени у границы `maxDays`.

2. **Потолок DOM-рендера.** `div/span` на бакет: `ширина × строки × realtime-тик` → десятки тысяч живых
   узлов, неизбежный jank при «сотнях инструментов на записи». *Release:* WebGL2 через `regl` (инстансинг
   прямоугольников, zoom/pan через uniform). Триггер миграции — первый заметный jank, не раньше.

3. **Сессионная ось ⟂ плавный zoom.** Схлопнутая посессионная ось несовместима с линейной матрицей зума.
   *Release:* **два режима** — «Обзор» (схлопнутая, как сейчас) и «Анализ/зум» (линейное время, инерция).

4. **app-кэш vs continuous aggregates.** Свой кэш прост и ленив (наполняется только для видимых
   инструментов), но не самообновляется и живёт на прикладном уровне. *Release:* caggs с realtime-политикой
   Timescale. Замена — за `ITradeActivityStore`, контракт не меняется.

5. **Сессии per-instrument и таймзоны (в т.ч. US).** Единого «рабочего времени» нет: Finam даёт инструменты
   разных площадок (MOEX/срочка/валюта/US с ET+DST). Нужна абстракция «торговая сессия площадки» с
   **IANA-таймзоной**. Источник расписания — открытый вопрос (статус бумаги от TRANSAQ vs эмпирика
   `md_trade` vs статический календарь 7c). Влияет на честность фона (7h) и авто-запись (7i).

6. **Креды в памяти.** `ICredentialStore` — in-memory: при рестарте хоста подключение теряется, а открытые
   сегменты/интервалы «осиротевают». *MVP-смягчение:* recovery на старте (7h.0) закрывает осиротевшее
   честно. *Release:* персист кред/подключений (пересекается с phase 10 `user_settings`/секреты).

7. **Присутствие vs интенсивность.** Слой сделок сейчас качественный (есть/нет) — так и задумано для MVP.
   *Release (опц.):* интенсивность (плотность сделок → насыщенность) добавляется тем же `count`/`bool_or`
   в роллапе, без смены модели.

## Что можно улучшить уже сейчас (без release-работ)

- Косметика «разрыва разрешения»: keep-previous-while-loading + гистерезис переключения ступени (п. 1).
- Ручной seed `capture_liveness` с зазором → визуальная проверка красного разрыва до автоматики (7h.7).
