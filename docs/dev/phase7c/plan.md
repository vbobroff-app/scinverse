# Phase 7c. Реальное расписание торгов MOEX (ISS)

Замена зашитого `MoexSchedule` (эвристика «будни 08:50–23:50 / выходные 10:00–19:00») на **реальные
данные Мосбиржи** через публичный ISS API: производственный календарь (праздники, рабочие выходные,
дни ДСВД) и внутридневное расписание сессий по рынкам. Ссылки и разбор endpoint'ов — в
[apply.md](apply.md); статус — в [report.md](report.md).

**Статус:** `MVP DONE` (2026-07-12). **Stage:** 1. **Зависимости:** phase 7b (`MoexSchedule`,
`/api/sessions`, `TimeframePanel`, ось `TimeAxis`). Ядро 7c.1–7c.7 + 7c.10 готово; 7c.8 (новости) и
7c.9 (статус борда в карточке) осознанно отложены за MVP — см. [report.md](report.md).

## Мотивация

В phase 7b границы сессии зашиты константами, а торговые дни выводятся из наличия `md_trade`. Этого
достаточно для демо, но:

- праздники/сокращённые предпраздничные дни и «рабочие выходные» не учитываются;
- дни ДСВД (сессия выходного дня) не каждые выходные — расписание утверждает биржа;
- часы у ФР и СР различаются и меняются (напр. переход ЕТС 23.03.2026);
- ось `D1` должна показывать реальные границы и «круглые» промежуточные времена именно текущей сессии.

Мосбиржа отдаёт всё это публично (без авторизации, JSON/XML) через ISS. Берём как источник истины,
`MoexSchedule` оставляем фолбэком при недоступности ISS.

## Область (in scope)

- **7c.1 ISS-календарь движка (БЕСПЛАТНО, `/iss/engines/{engine}`).** Источник — `timetable` (недельное
  расписание) + `dailytable` (исключения по датам: праздники/переносы/сокращённые дни, вкл. будущие),
  один запрос `?iss.only=timetable,dailytable&iss.meta=off&lang=ru`. НЕ платный `/iss/calendars`.
  Добавляем `IExchangeCatalog.GetEngineCalendarAsync(engine)` (кэш в памяти) поверх готового
  `IssExchangeCatalog`/`IssTable`. Разбор — в [apply.md §2](apply.md).
- **7c.2 Модель календаря (Domain).** `EngineCalendar` (чистая, тестируемая): для даты → торговый день?
  + внешние границы (`open`/`close`, МСК), приоритет `dailytable` над `timetable`. `IMarketCalendar`
  (`ShapeSessionsAsync`) + чистый `TradingCalendar.Shape`: часы дней — дат-точные из календаря
  (сокращённые/регламентные, напр. FORTS 06:50–23:50 с 14.07.2026), праздники (`is_work_day=0`)
  отбрасываются; при недоступности ISS — фолбэк на эвристику `MoexSchedule` (лог warn). **Выходные
  (ДСВД):** внешнее окно движка (все 7 дней) слишком широкое (это общая доступность), поэтому для
  сб/вс берём узкое окно `MoexSchedule` (09:50–19:00), а сам факт торгов выходного — по нашим данным
  (`QueryTradingDaysAsync`), т.к. `dailytable` не перечисляет ДСВД-выходные.
- **7c.3 Кэш (in-memory, персистентность → Phase 13).** `GetEngineCalendarAsync` кэшируется в
  `IMemoryCache` (TTL ~12ч), ISS дёргается не на каждый запрос. Персистентный кэш в БД + инвалидация
  по обновлению — вынесены в сквозную [Phase 13 «Кэширование»](../phase13/plan.md) (таблицы
  `trading_calendar`/`market_session` — там же). Отдельная миграция в 7c не нужна.
- **7c.4 Интеграция с `/api/sessions` и `/api/coverage/extent`.** Сессии строятся из календаря
  (праздники исключены, ДСВД включены по расписанию, реальные часы), а не из эвристики; `market`
  выбирается по инструменту (FORTS → `futures`, акции → `stock`).
- **7c.5 Страница «Биржи → Структура».** Отдельный раздел админ-фронта: дерево `движок → рынок →
  борд`, а на листе борда — **актуальный список торгуемых инструментов** (из ISS `securities`
  соответствующего борда). Backend-прокси `GET /api/exchanges/*` (engines/markets/boards/securities)
  с редким кэшем; UI — read-only справочник (поиск, фильтр по борду). ISS отдаёт и статику, и
  `marketdata` (котировки с задержкой) — на этой странице достаточно статики.
- **7c.6 Фронт (таймлайн).** `moexSession.ts` использует границы из `SessionDto` как есть (уже так);
  убрать клиентскую эвристику часов, оставить только форматирование. Ось `D1` уже показывает start/end
  + круглые метки (сделано в предыдущем шаге).
- **7c.7 Тесты.** unit (парсинг ISS-ответов из зафиксированных фикстур, маппинг в `MarketSchedule`,
  fallback при ошибке), integration (кэш `trading_calendar`), опц. «живой» smoke к ISS под флагом.

- **7c.8 (задел) Лента новостей/событий.** Поллер ISS `sitenews`/`events` → `NewsEvent`
  (регламенты, статусы, уведомления вроде смены расписания FORTS). Push-канал коннектора
  (TRANSAQ `news_header`/`news_body`, QUIK-сообщения) — отдельным инкрементом. Детали — в
  [apply.md §3d](apply.md).

- **7c.9 Статус инструмента в карточке (двухслойный).** В строке инструмента каталога — авторитетный
  статус торгов, решающий кейс «биржа открыта, но инструмент не торгуется» (напр. суббота: доступ
  есть, а борд вне ДСВД). Два слоя:
  - **Слой A — расписание борда (авторитетно, для всех инструментов, НЕ lagging).** `Открыто` /
    `Закрыто` / `Пре-опен` (аукцион) — из `session_schedule` по `(boardid, type, time_from/time_till)`
    на текущий момент МСК. Не зависит от потока данных; работает и для незаписываемых инструментов.
    В субботу: инструмент борда с ДСВД → `Открыто`, борда без ДСВД → `Закрыто`.
  - **Слой B — активность записи (только для записываемых, lagging-доп.).** Для инструментов с
    активной записью — `active`/`waiting` по времени последней сделки с **адаптивным порогом ~30с**
    (учёт разной ликвидности: неликвидные торгуются редко и при открытых торгах). После 30с тишины —
    показываем не бинарный статус, а info «последняя сделка N сек/мин назад». Источник —
    `coverageExtended.to` / сегмент покрытия (данные уже есть на фронте).

  Приоритет отображения: авторитетный слой A задаёт основной статус борда; слой B — уточнение
  «реально ли капают сделки» поверх, только для записываемых. Статичный `instrument.active` (справочный
  «торгуемый в принципе») остаётся отдельным атрибутом, не путать со статусом борда.

- **7c.10 Категоризация фьючерсов (справочник `futures_asset_class`).** Категория фьючерса = класс
  базового актива («Группа контрактов» по спецификации MOEX `s205`): Индексы/Акции/Валюта/Ставки/
  Товары. Курируемый справочник `asset_code → category`, наполняемый **из ISS по кнопке** (без
  таймера): рефреш собирает актуальные `ASSETCODE` FORTS-фьючерсов, авто-классифицирует (сид-карта
  s205 + резолв спот-актива через `/iss/securities`), новые/неизвестные помечает «на проверку»
  (`confirmed=false`) для ручного курирования. Категория затем питает динамические фильтры-плашки
  (как 7d). Детали и таксономия — в [apply.md §3f](apply.md).

## Вне области (out of scope)

- Биржевой флаг приостановки **конкретной бумаги** (`sec_status` / `suspended`) — отложен (отдельно);
  статус по **расписанию борда** — уже в scope (7c.9), это разные вещи.
- Расчётная/клиринговая сессии (`settlement_session`/`clearing_session`) — читаем, но пока не
  используем в UI.
- Платные/real-time данные ISS, стаканы, история — не нужны.

## Критерии приёмки

1. `GET /api/sessions` отдаёт часы дня **из ISS-календаря движка** (дат-точные: сокращённые/
   регламентные, напр. FORTS 06:50–23:50 с 14.07.2026), а не из зашитой эвристики; **праздники**
   (`is_work_day=0`) исключаются. Выходные (ДСВД) — по нашим данным, с узким окном ДСВД.
2. При недоступности ISS сервис прозрачно откатывается на `MoexSchedule` (лог warn + фолбэк), фронт живой.
3. Календарь движка кэшируется в памяти (TTL); повторные запросы не бьют в ISS (персистентный кэш —
   Phase 13).
4. `dotnet build` 0/0; unit (парсинг/fallback) и integration (кэш) зелёные; `tsc`/`vitest` зелёные.
5. Карточка инструмента показывает статус борда (`Открыто`/`Закрыто`/`Пре-опен`) из расписания: в
   субботу инструменты бордов с ДСВД → `Открыто`, без ДСВД → `Закрыто` (при живом подключении). Для
   записываемых — `active`/`waiting` с порогом 30с, дальше info «последняя сделка N назад».
