# Phase 7d. Динамические фильтры каталога (admin frontend)

Расширение phase 7 (админ-фронт): каталог инструментов получает **динамические фильтры-плашки**
(chips) в шапке карточки провайдера вместо статичного дропдауна категории + чекбокса. Дизайн
Stage 1 — в [../apply.md](../apply.md); особенности реализации — в [apply.md](apply.md); статус — в
[report.md](report.md).

**Статус:** `PLANNED`. **Stage:** 1. **Зависимости:** phase 7 (`FilterBar`, `OhsStore`,
`/api/instruments`), phase 6c (категории/деривативы).

## Мотивация

Сейчас фильтр каталога — фиксированный ряд: дропдаун категории (`CategoryDropdown`), строка поиска,
чекбокс «только запущенные», счётчик. Это не масштабируется: фильтров будет больше (условия записи,
биржи, выделенное), а Finam отдаёт инструменты с множества площадок. Нужен паттерн
«добавляемых плашек» (как в каталоге моделей проекта `mars`): пользователь сам набирает нужный
набор фильтров, каждый — своя плашка со своим поповером условий.

## Целевой UX

Ряд управления фильтрами (шапка карточки провайдера):

```
[Инструменты ×][Выбор ×][Биржи ×]  [+][×]                         [🔍 Поиск …]
└──────────── активные плашки ─────┘ └ добавить/сбросить ┘         └ поиск справа ┘
```

- **`[+]`** — открывает меню со списком доступных фильтров (галочка у уже добавленных). Выбор пункта
  добавляет/убирает соответствующую плашку. Порядок плашек = порядок добавления.
- **`[×]`** — сбрасывает все активные фильтры (плашки исчезают), поиск не трогает.
- **Плашка** — `[Название ×]`. Клик по телу открывает поповер условий фильтра; `×` убирает плашку.
  Если у фильтра выбрано значение — оно дописывается в подпись (`Инструменты: Фьючерсы`) и плашка
  подсвечивается.
- **Поиск** — всегда виден **справа**, отдельным полем (короче прежнего), с иконкой-лупой. В плашки
  не входит.

### Набор фильтров v1

| Плашка | Тип поповера | Значение | Бэкенд |
| ------ | ------------ | -------- | ------ |
| **Инструменты** | одиночный выбор категории | `Все / Фьючерсы / Акции / Валюта / Облигации / Индексы` | `category` (есть) |
| **Выбор** | чек-лист условий | `Запущенные` / `Не пустые` / `Выделенные` (любая комбинация) | `onlyRecording` (есть) + `nonEmpty` (нов.) + `instrumentIds` (нов.) |
| **Биржи** | чек-лист бирж | `MOEX` (пока единственная; задел под площадки Finam) | `exchanges` (нов., задел) |

Условия «Выбор» комбинируются по И (AND): «Запущенные + Выделенные» = запущенные среди выделенных.

## Область (in scope)

- **7d.1 Backend: расширение `InstrumentQuery`.** Поля `NonEmpty` (bool), `InstrumentIds`
  (`IReadOnlyList<long>?`), `Exchanges` (`IReadOnlyList<string>?`). SQL-фильтры в `InstrumentStore`:
  - `NonEmpty` → `EXISTS (coverage_segment cs WHERE cs.instrument_id = i.instrument_id)` (был хоть
    один сегмент записи, т.е. есть данные).
  - `InstrumentIds` → `i.instrument_id = ANY(@instrumentIds)` (для «Выделенных»).
  - `Exchanges` → `i.board_id = ANY(@boards)`, где `@boards` — объединение бордов выбранных бирж
    (резолвер `ExchangeBoards`, зеркало клиентского `exchange.ts`; пока MOEX ⇒ все борды).
- **7d.2 API.** `GET /api/instruments` — новые query-параметры `nonEmpty`, `instrumentIds` (повтор.
  или CSV), `exchanges` (повтор. или CSV). Контракт `IOhsApi` + `OhsApiClient` (тесты) синхронно.
- **7d.3 core: модель фильтров.** В `OhsStore` — `activeFilters$: BehaviorSubject<FilterKey[]>`
  (порядок плашек) + расширение `instrumentQuery$` полями `nonEmpty`, `instrumentIds`, `exchanges`.
  Методы: `addFilter/removeFilter/clearFilters`, `setCategory`, `setSelectionConditions`,
  `setExchanges`. `instrumentIds` для «Выделенных» вычисляется из `selectedInstruments$`.
- **7d.4 api.ts (frontend).** `buildInstrumentsQuery` — сериализация новых полей; типы в `types.ts`.
- **7d.5 UI: компоненты плашек.** `FilterChips` (ряд плашек + `[+]` меню + `[×]`), поповеры:
  `CategoryFilterMenu` (одиночный), `SelectionFilterMenu` (чек-лист условий), `ExchangeFilterMenu`
  (чек-лист бирж). Переиспользуемый примитив поповера `FilterPopover` (клик-вне-закрытие, портал).
- **7d.6 UI: рефактор `FilterBar`.** Раскладка `[FilterChips] … [Search]`: плашки слева, поиск
  справа (короче, иконка-лупа). Удаление `CategoryDropdown` из ряда и чекбокса «только запущенные»
  (их роль берут плашки). Счётчик «Найдено: N» сохраняем.
- **7d.7 Тесты.** unit backend (`FakeInstrumentStore` + новые фильтры), api-тест новых параметров,
  vitest на модель фильтров (`OhsStore`: add/remove/clear, маппинг условий в query), компонентные
  тесты `FilterChips` при необходимости.

## Вне области (out of scope)

- Реальная мультибиржа (несколько бирж в данных) — сейчас всё MOEX; фильтр «Биржи» — задел
  (end-to-end плумбинг + резолвер), функционально no-op до появления не-MOEX инструментов.
- Персист набора фильтров в `localStorage` (как в `mars`) — возможный follow-up.
- Фильтры по датам/диапазонам значений, произвольные операторы — не нужны для каталога.
- Server-side пагинация «Выделенных» большого объёма — набор выделенного мал (ручной выбор).

## Критерии приёмки

1. `[+]` добавляет плашку из списка (Инструменты / Выбор / Биржи), `×` на плашке убирает её,
   `[×]` сбрасывает все; поиск при этом сохраняется.
2. «Инструменты» фильтрует по категории; «Выбор» комбинирует `Запущенные`/`Не пустые`/`Выделенные`
   по И; «Биржи» плумбится до бэкенда (MOEX).
3. Backend: `nonEmpty`, `instrumentIds`, `exchanges` корректно сужают выборку и `Total`
   (`COUNT(*) OVER()` до LIMIT).
4. Поиск справа отдельным полем с лупой; счётчик «Найдено: N» корректен под активными фильтрами.
5. `dotnet build` 0/0; unit/api зелёные (добавлены новые); `tsc --noEmit` + `vitest` зелёные.
