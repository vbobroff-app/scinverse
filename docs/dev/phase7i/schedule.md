# Идеология управления расписанием (Schedule)

**Статус:** дизайн зафиксирован (phase 7i). Это опорный документ: он фиксирует *как система думает о
торговом расписании* и структуру таблиц. Реализация и её срез — в [plan.md](plan.md) / [apply.md](apply.md).

> **Весь слой расписания = две таблицы** (`market_schedule` + `market_schedule_exception`) с общей моделью
> scope и единым резолвером. Вокруг них — 3 куска кода: `scopeOf(instrument)`, резолвер (Гант + Supervisor),
> daily pre-flight (сверка с API → создание исключений). Новых сущностей больше не требуется.

## Зачем это нужно

Расписание — фундамент для двух вещей:

1. **Настройка авто-записи** (Supervisor, 7i): автомат должен знать, *когда* инструмент торгуется,
   чтобы вооружать/разоружать запись и не растить «нечестный» фон вне сессии. И — не вставать на запись
   в день, когда торгов нет (праздник, траур, инцидент).
2. **Отрисовка Ганта**: «колбаски» покрытия и подложка сессии рисуются по регламенту, **действовавшему
   на ту дату** (а не по сегодняшнему), включая исторические данные (бэкфилл QScalp/Finam).

Обе задачи спрашивают одно и то же: **«какое расписание действовало/действует для инструмента X на дату D?»**

## Главный принцип

> **Единый источник истины — наши таблицы в БД. Внешний API — не источник, а подтверждатель.**
> Он умеет *предложить* правку (сформировать исключение), но **финальное решение всегда за пользователем.**

Это защищает курируемые данные от порчи автоматом и не ставит нас в зависимость от чужого аптайма для
«просто записать обычный день».

## Модель scope (общая для обеих таблиц)

Обе таблицы адресуют данные одинаковой иерархией «сверху вниз» (от общего к частному). Заполнены только
те уровни, до которых спускается правило; ниже — `NULL` (= «на всё внутри»). В данных — **коды** (латиница),
как в справочнике `futures_asset_class.category`; русские подписи — только в UI (`FuturesAssetTaxonomy.CategoryTitles`).
Значения `sec_type`/`category` **зависят от рынка** (per-market словари) — глобального `CHECK` на них нет.

| Уровень | Колонка | Срочный (derivatives) | Фондовый (stock) | Валютный (currency) |
|---|---|---|---|---|
| Рынок | `market` (NOT NULL) | `derivatives` | `stock` | `currency` |
| Вид | `sec_type` | `futures` \| `options` | `shares` \| `bonds` \| `dr` … (TBD) | `td` \| `tm` … (TBD) |
| Категория | `category` | `currency` \| `shares` \| `index` \| `commodity` \| `rate` \| `other` | (TBD) | (TBD) |
| Инструмент | `instrument` | SECID | SECID | SECID |

> Значения `category` **совпадают** с `futures_asset_class.category` (`currency/shares/index/rate/commodity/other`) —
> чтобы `scopeOf` мапил категорию напрямую, без перевода. Коды `market` (`derivatives/stock/currency`) намеренно
> отличаются от `sec_type` (`futures/options`), чтобы «срочный рынок» и «фьючерс» не схлопнулись в одно слово.

**Важно:** в расписании хранится не полная таксономия инструментов, а только те уровни, **где реально
расходится расписание**. Полный справочник инструментов живёт отдельно; scope здесь — грубая проекция.
Пока у фондового/валютного расписание одно на весь рынок — их строки живут на уровне `market`
(под-уровни `NULL`), детализируем по мере находок.

## Два слоя данных

### 1. Базовое расписание — `market_schedule` (V013/V014 + расширение V016)

Структурный, **версионный** (SCD-2) распорядок «как **обычно** выглядит торговый день». Держит **и будни,
и выходные в одной строке** (`wd_*` + `we_*`) + фазы (JSONB); `we_* = NULL` → в выходные не торгует.

- Scope: `(market, sec_type, category, instrument)`; версия — `effective_from`.
- Действующая на дату `D` = **самая специфичная** подходящая строка с `effective_from ≤ D`
  (приоритет `instrument > category > sec_type > market`, затем `effective_from` DESC).
- Меняется **редко** (единицы событий: ДСВД 01.03.2025, ЕТС 23.03.2026, расширение 06:50 с 14.07.2026) →
  строк мало, наполняем вручную/исследовательски и через «повышение» исключений (см. ниже).

Структура (после V016):

```sql
market         TEXT NOT NULL   -- derivatives | stock | currency (срочный | фондовый | валютный)
sec_type       TEXT            -- «вид», per-market коды: futures|options|shares… (NULL = любой)
category       TEXT            -- «категория», коды как в futures_asset_class: currency|shares|index|… (NULL = любая)
instrument     TEXT            -- SECID (NULL = не привязано; для уникальных, напр. новый BTC-фьюч)
effective_from DATE NOT NULL   -- версия действует с этой даты
wd_open, wd_close TIME NOT NULL -- будни: границы дня (вкл. аукцион)
we_open, we_close TIME          -- выходные (ДСВД); NULL = не торгует
phases         JSONB           -- {"weekday":{фазы}, "weekend":{фазы}}
confidence     TEXT NOT NULL DEFAULT 'assumed'  -- authoritative | empirical | assumed
source, note   TEXT
-- UNIQUE (market, coalesce(sec_type,''), coalesce(category,''), coalesce(instrument,''), effective_from)
```

### 2. Исключения по дате — `market_schedule_exception` (новое, V017)

Отклонения **на конкретную дату** от базового: праздник (торгов нет), сокращённый день, сдвиг часов.
Отдельная таблица (паттерн ISS `timetable` vs `dailytable`); **никогда не мутирует base**. Обычно создаётся
**авто** после сверки с API (см. pre-flight), scope чаще всего = инструмент.

Дата — это один день (будни или выходной сами по себе), поэтому деления `wd_*/we_*` нет — только окно дня:

```sql
exc_date       DATE NOT NULL   -- одна дата
market         TEXT NOT NULL   -- тот же набор scope-колонок, что и в base
sec_type, category, instrument TEXT
kind           TEXT NOT NULL   -- no_trade | shifted | shortened
open_time, close_time TIME      -- для shifted/shortened; NULL при no_trade
phases         JSONB           -- фазы этого дня (если сдвиг)
confidence     TEXT NOT NULL DEFAULT 'assumed'
source         TEXT            -- 'ISS dailytable' | 'Finam Schedule' | 'user'
resolved       BOOLEAN NOT NULL DEFAULT FALSE  -- пользователь разобрал → сверка не спамит
note           TEXT
-- UNIQUE (market, coalesce(sec_type,''), coalesce(category,''), coalesce(instrument,''), exc_date)
```

> `kind`: `no_trade` — не торгует (праздник/траур); `shifted` — иные границы/фазы; `shortened` — ранний
> клоуз (частный `shifted`, отдельная семантика для UI/уведомлений).

## Две проекции решения пользователя

Когда пользователь разбирает исключение, он выбирает правку в **двух ортогональных проекциях**:

1. **По времени** → *какие столбцы профиля* правим: `будни` (`wd_*` + будние фазы) / `выходные`
   (`we_*` + выходные фазы) / `любой день` (оба). Границы и фазы неразрывны — правятся вместе.
2. **По scope** → *на каком уровне* закрепляем: `этот инструмент` / `все валютные фьючерсы`
   (`sec_type=futures, category=currency`) / `все опционы` (`sec_type=options`) / `весь рынок`.

**Одно решение = upsert одной base-строки** на `(выбранный scope, effective_from = дата)`, где обновляются
только столбцы выбранной временной проекции. `[Оставить разовым исключением]` = не повышаем, `exclude`
остаётся на уровне инструмент+дата.

## Единый резолвер

Один код отвечает на «расписание для X на D» и обслуживает **и Гант, и Supervisor** (чтобы отрисовка и
реальное окно записи не разошлись):

```
resolve(instrument, D):
  scope  = scopeOf(instrument)                 # (market, sec_type, category) из справочников
  base   = market_schedule: самая специфичная строка (scope+instrument), effective_from <= D
  window = isWeekend(D) ? base.we_* : base.wd_*   # we_* = NULL → в этот выходной CLOSED
  exc    = market_schedule_exception: самая специфичная строка на D (scope+instrument)
  if exc == null:          return window                # обычный день
  if exc.kind == no_trade: return CLOSED                # торгов нет
  else:                    return exc.open/close/phases # окно дня заменено отклонением
```

**Поведение незарезолвенного исключения:** `resolved` **не влияет** на резолвер — `exclude` действует всегда
(источник авторитетный, создан по подтверждению API). `resolved=false` — лишь маркер «пользователь ещё не
решил, повышать ли в base» + анти-спам: в пределах даты дубликат не создаётся (UNIQUE) и уведомление не
перевыпускается. Систематическое расхождение на следующий день даёт новый `exclude` + уведомление — это
подталкивает повысить правило до base.

## Примеры заполнения

**`market_schedule` (base):**

| market | sec_type | category | instrument | effective_from | wd | we | confidence | смысл |
|---|---|---|---|---|---|---|---|---|
| derivatives | futures | — | — | 2026-07-14 | 06:50–23:50 | 10:00–19:00 | authoritative | общий регламент фьючерсов FORTS |
| derivatives | futures | currency | — | 2026-07-14 | 06:50–23:50 | **NULL** | authoritative | валютные фьючи не торгуют в выходные (специфичнее — выигрывает) |
| derivatives | futures | commodity | `BR-1.27` | 2026-07-14 | 07:00–23:50 | 10:00–19:00 | empirical | уникальный инструмент со своим окном |
| stock | — | — | — | 2025-03-01 | 06:50–23:50 | 09:50–19:00 | authoritative | фондовый живёт на уровне рынка |

**`market_schedule_exception` (exclude):**

| exc_date | market | sec_type | category | instrument | kind | окно | source | resolved |
|---|---|---|---|---|---|---|---|---|
| 2026-01-01 | derivatives | — | — | — | no_trade | — | ISS dailytable | true |
| 2026-07-14 | derivatives | options | currency | `Si78000BF6` | shifted | 10:00–24:00 | Finam Schedule | false |
| 2026-03-07 | derivatives | — | — | — | shortened | 06:50–18:45 | ISS dailytable | true |

Резолвер для `Si78000` на 14.07.2026: `scope=(derivatives, options, currency)` → базовый профиль (вторник, будни),
затем exclude‑строка `shifted` перекрывает окно → итог **10:00–24:00** только по этому инструменту и только
в этот день. Гант рисует по нему, авто пишет с 10:00.

## Источники подтверждения (и кто что умеет)

Проверено на живом API (13–15.07.2026). Подтверждатель — **не один**, и у каждого своя область
компетенции и свои слепые зоны. База (курируем руками из регламента MOEX) остаётся источником истины;
внешние источники лишь подтверждают/предлагают исключения.

| Источник | Что даёт | Гранулярность | Горизонт | Auth |
|---|---|---|---|---|
| **Finam `Schedule`** | фазы дня + клиринг (`OPENING_AUCTION/EARLY/CORE/LATE_TRADING/CLEARING/CLOSED`) | per-instrument (SECID@MIC), но по сути **шаблон рынка** | сегодня + ~2 дня вперёд | секрет → JWT |
| **ISS `session_schedule`** | фазы дня (`oa_booking/oa_pricing/morning/main/evening/settlement/clearing`) | **market-wide** (`secid="-"`) | **только сегодня** (игнорит `date=`) | нет (публично) |
| **ISS `dailytable`** | праздники/переносы/сокращённые (внешняя граница дня `is_work_day, start/stop`) | market-wide (движок) | **2018 → будущее** | нет (публично) |
| **ISS `timetable`** | недельный шаблон (внешняя граница дня по дню недели) | market-wide (движок) | шаблон | нет (публично) |

### Ключевая находка (15.07.2026): слепая зона обоих «фазовых» источников

И **Finam `Schedule`, и ISS `session_schedule` — market-wide по внутридневным фазам** и **не различают
per-category карв-ауты**. Живой кейс: валютный опцион `Si80000BG6` (15.07) — оба источника показывают
`morning 07:00–10:00`, но по факту (QUIK, 07:19 МСК: *«Сейчас эта сессия не идёт»*) валютные опционы в
утреннюю сессию **не торгуются** и стартуют в 10:00 (регламент MOEX, новость `moex.com/n101980`:
«в утренний период — все инструменты, **за исключением опционов на валютные пары**»).

**Вывод:** правило «валютные опционы стартуют в 10:00» есть **только в регламенте** → его источник —
**наша курируемая base** (`derivatives/options/currency`, V019). Ни Finam, ни ISS его не знают. Поэтому
pre-flight по такому инструменту закономерно даёт `shifted`-исключение (база 10:00 vs источник 07:00),
и правильный резолв — **в пользу base** (человек отклоняет exclude).

> Аналогично `timetable` фьючерсного движка отдаёт `week_day=6 → is_work_day=1, 07:00–23:59:59`
> («суббота торгуется»), хотя валютный фьючерс по выходным закрыт — правило выходных тоже **не движковое**.

### Роли источников (кто на что годится)

- **Праздники / сюрприз-выходные / сокращённые дни** → **ISS `dailytable`** (история + будущее, бесплатно).
  Основной детектор `no_trade`/`shortened` уровня рынка.
- **Ежедневная сверка «сегодня рынок открылся как в base?»** → **ISS `session_schedule`** (чище по составу
  фаз: есть `settlement`/`clearing` раздельно) **или** **Finam `Schedule`**. По внешним границам рынка
  эквивалентны (см. таблицу сопоставления фаз ниже).
- **Per-instrument near-term подтверждение** → **Finam `Schedule`** (единственный per-SECID; но помним про
  слепую зону карв-аутов и короткий горизонт).
- **Per-category / исторические тонкие фазы** → **никто из API**; только курируемая base по регламенту
  (события смены регламента редки: ДСВД 01.03.2025, ЕТС 23.03.2026, расширение 06:50 с 14.07.2026).

Сопоставление фаз (срочный рынок, один-в-один по границам, отличаются только названия/подача):

| Фаза | ISS `session_schedule` | Finam `Schedule` |
|---|---|---|
| Аукцион открытия | `oa_booking` + `oa_pricing` 06:50–06:59:50 | `OPENING_AUCTION` 06:50–06:59:50 |
| Утренняя | `morning_session` 07:00–10:00 | `EARLY_TRADING` 07:00–10:00 |
| Основная | `main_session` 10:00–19:00 | `CORE_TRADING` 10:00–19:00 |
| Вечерняя | `evening_session` 19:00–23:50 | `LATE_TRADING` 19:00–23:50 |
| Расчётная/клиринг | `settlement_session`/`clearing_session` (раздельно) | `CLEARING` |
| «Закрыто» (ночь) | — (не перечисляет) | `CLOSED` (явно) |

## Абстракция подтверждателя (`IScheduleConfirmer`) и Интеграции — реализовано

MOEX ISS добавлен **вторым адаптером-интеграцией** рядом с Finam. Pre-flight больше не прибит гвоздями к
Finam: общий контракт + адаптеры, ключуемые по `external_service.adapter` (реестр `IScheduleConfirmerRegistry`):

- **`IScheduleConfirmer`** (`Domain/Schedule`, `Adapter`: `finam` | `moex-iss` | …):
  `RequiresSecret` (bool), `ProbeAsync(query)` (health-check) и `GetScheduleAsync(query)` →
  **нейтральный `ConfirmerSchedule`** (список `ConfirmerSession` с нормализованным `ScheduleSessionKind`
  {Auction, Trading, Clearing, Settlement, Closed}, а не сырой `FinamSchedule`).
- **`ConfirmerQuery`** несёт и `Symbol` (Finam: `SECID@MIC`), и `Engine` (ISS: futures/stock/currency), и
  `Date`, и `Secret` — каждый адаптер берёт своё. `SchedulePreflight` строит запрос по scope инструмента,
  а окно дня сводит общий хелпер `ScheduleWindow.Trading` (open/close по Auction+Trading сессиям в МСК).
- **Адаптеры:** `FinamScheduleConfirmer` (обёртка `IFinamApi`, `RequiresSecret=true`), `IssScheduleConfirmer`
  (обёртка `IExchangeCatalog.GetSessionScheduleAsync` над `session_schedule`, `RequiresSecret=false`).
- **Capabilities** источника (`IScheduleConfirmer.Capabilities`):
  - `schedule` — фазы/окно дня (Finam: per-instrument; ISS: market-wide, сегодня) — **реализовано**;
  - `calendar` — праздники/переносы по дате (ISS `dailytable`, история + будущее; Finam — нет) —
    **реализовано** через `ICalendarConfirmer` (реализует ISS). Pre-flight, если назначенный источник
    calendar-capable, сканирует горизонт вперёд (14 дн.) и на нерабочие дни, где база ожидает торги,
    заводит `no_trade`-исключение на уровне РЫНКА (`market`-scope) — то, что `session_schedule`/Finam
    (сегодня/~2 дня) не видят. В UI интеграции — карточка «Календарь (исключения)».
- **No-auth адаптеры.** ISS публичный → `RequiresSecret=false`; форма интеграции и endpoint `probe`/`schedule`
  допускают **пустой секрет**, а pre-flight запрашивает секрет только у auth-адаптеров.
- **Символ — деталь адаптера.** `SECID@MIC` нужен только Finam; ISS адресуется движком. Общий интерфейс
  оперирует `ConfirmerQuery` (symbol|engine + date), а не «символом источника».

### Эволюция назначения источника: `use_for_schedule` → source-mapping

Сейчас «кто подтверждает» — булев эксклюзивный флаг `external_service.use_for_schedule` (ровно один
источник, V018). Идею маппинга **`system_source(capability, market, service_id)`** реализуем **без
изменения схемы** — через композитную интеграцию **«Scinverse API»** (adapter `scinverse`, `ScinverseScheduleConfirmer`,
V022): это фасад над Finam/ISS, который сам роутит запрос по capability/движку (cross-source). Галку
`use_for_schedule` ставим на неё — реестр/pre-flight/эндпоинты не меняются (резолв по adapter).

**Инкапсуляция:** наружу фасад — единый **«Scinverse»**; внутренний источник (сегодня ISS/Finam) —
деталь реализации, которая не протекает в контракт (`/integrations/{id}/schedule`|`/calendar`) и в UI.
Поэтому карточки помечены **«Scinverse schedule»** / **«Scinverse dailytable»**, а не именем провайдера:
если завтра Finam начнёт отдавать календарь, меняем только роутинг внутри фасада — интерфейс тот же.

**Нормализация «в наш формат» на приёме.** Внешние источники (ISS, Finam) отдают *инклюзивный* конец
фазы на секунде `:59` (напр. `09:59:59` = «до 10:00», конец суток — `23:59:59`). Фасад приводит времена к
нашему канону — полуинтервал `[start, end)` в целых минутах: **Start → floor до минуты, End/Close → ceil до
минуты**. Итог: `09:59:59 → 10:00`, `18:59:59 → 19:00`, `23:59:59 → 00:00` (следующие сутки), а фазы стыкуются
без «щелей». Логика живёт в эндпоинтах фасада (`NormalizeToHouseFormat` для расписания, `ComposeCalendarDay`
для часов дня) — наружу времена уже канонические.

**Роутинг фасада `scinverse`:**
- **Календарь** (праздники/переносы) — у фасада **собственный эндпоинт** `GetCalendarAsync`, где живёт
  композиционная логика «какое поле откуда брать» (метод `ComposeCalendarDay`, точка расширения под
  cross-source правила). Сейчас базовый и единственный источник дней — **ISS** `dailytable`; наружу — как
  капабилити самого Scinverse. Когда данные появятся у Finam/другого источника — правило дописывается
  внутри `ComposeCalendarDay`, контракт `/calendar` и UI не меняются.
- **Schedule `futures`** — **ISS** (бесплатно, market-wide); при пустом/ошибке — **Finam** (fallback).
- **Schedule `stock`/`currency`** — **Finam** (per-instrument; ISS `session_schedule` по ним **пуст**);
  при недоступности — ISS (fallback).
- Секрет для Finam-ветки фасад берёт у назначенного `finam`-сервиса (сам фасад `RequiresSecret=false`);
  если Finam не настроен — деградирует на ISS. Зависит от **конкретных** адаптеров, не от реестра
  (иначе цикл в DI). Роутинг покрыт `ScinverseScheduleConfirmerTests`.

**Ограничение Finam Schedule:** окно короткое (сегодня + ~2 дня) и без истории → near-term подтверждатель,
не детектор далёких праздников. Праздники — за ISS `dailytable`.

## Достоверность (`confidence`)

- `authoritative` — офиц. MOEX (новость/приказ/s1167/ISS dailytable) или Finam `Schedule`.
- `empirical` — реконструкция по свечам (Finam Bars RI/Si/SBER: границы дня + клиринг → фазы).
- `assumed` — предположение.

Уточнение = добавление строки с более высокой достоверностью (старую не трогаем — история сохраняется).

## Поток подтверждения (daily pre-flight)

```
раз в торговый день, ПЕРЕД арматурой записи, по scope-представителю (не по каждому инструменту — rate-limit):
  confirm = ISS dailytable(date) + Finam Schedule(instrument)
  if confirm ≈ base:            → ничего, пишем как обычно
  if confirm ≠ base:            → market_schedule_exception(date, scope, source, confidence, resolved=false)
                                  → резолвер (Гант + Supervisor) сразу учитывает исключение
                                  → уведомление severity=Warning (центр уведомлений, phase 11)
пользователь кликает уведомление → форма (две ортогональные проекции, см. выше):
  Применить к:  ( ) будням   ( ) выходным   ( ) любому дню          # временная проекция → wd_* / we_* / оба
  На уровне:    ( ) этот инструмент  ( ) вся категория  ( ) весь вид  ( ) весь рынок   # scope-проекция
  [ Применить ]                 = upsert base-строки на (scope, effective_from = дата) — правило постоянное
  [ Оставить разовым ]          = exclude остаётся на инструмент+дата (праздник/траур), resolved=true
```

## Политики надёжности

- **Fail-open при недоступности API.** Finam недоступен / токен протух / rate-limit → **не блокируем**
  запись, идём по базовому с warning. «Не смог проверить» ≠ «торгов нет».
- **`no_trade` — самый рискованный класс** (ложное срабатывание = пропущенный торговый день). Для
  *отмены записи* требуем согласие двух источников (ISS `is_work_day=0` **и** Finam) либо уведомляем ДО
  снятия с записи. Сдвиг часов — применяем и уведомляем (риск ниже).
- **`resolved`-флаг** — пользователь решил по дате → повторные ежедневные проверки не спамят тем же.
- **Идемпотентность** — повторный pre-flight за день не плодит дубликаты исключений.
- **Scope-probe** — один представитель на категорию/рынок, не по каждому инструменту.

## Открытые вопросы

1. **Кого спрашиваем для подтверждения** — *абстракция реализована* (`IScheduleConfirmer` + реестр,
   capabilities `schedule` **и** `calendar`; адаптеры `finam` {schedule}, `moex-iss` {schedule, calendar};
   pre-flight выбирает адаптер по `source.Adapter` и, если он calendar-capable, дополнительно сканирует
   праздники на горизонт вперёд). Назначение источника — пока булев эксклюзивный
   `external_service.use_for_schedule` (ровно один, V018); галка в UI переключает источник между кандидатами.
   Следующий шаг — маппинг `system_source(capability, market, service_id)`, чтобы `calendar` можно было
   держать на ISS независимо от того, кто назначен на `schedule`. См. раздел «Абстракция подтверждателя».
2. **`scopeOf(instrument)`** — маппинг SECID → `(market, sec_type, category)`: собрать из существующих
   справочников (таксономия/`futures_asset_class` для категории фьючерсов, справочник бордов — для рынка/вида).
   Это единственный «клеевой» код, без которого резолвер не заработает.
3. **Per-market словари scope** — валидные значения `sec_type`/`category` на рынок (фондовый/валютный пока
   TBD). Пока свободный текст без `CHECK`; отдельный справочник заведём при детализации этих рынков.
   Категория фьючерсов уже курируется в `futures_asset_class` (кнопка «Актуализировать из ISS»).
4. **Охват справочника опционами** — `futures_asset_class` наполняется только фьючерсами (`GetFortsFuturesAsync`).
   Для категорийного scope опционов (напр. «валютные опционы») нужно расширить актуализацию на опционы
   (у опциона тот же ASSETCODE, что у базового фьюча → маппинг тот же). Не блокирует V016/V017.
5. **Хранение секрета внешнего сервиса** — пока на бэке (см. plan «Интеграции»), после Keycloak (phase 10)
   пересмотреть.
