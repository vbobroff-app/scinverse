# Phase 7i — «Управление записью»: расписание автозаписи (Scheduler/Supervisor)

**Статус:** `IN PROGRESS`. Зависимости: **7h DONE**, **7c** (календарь ISS).
Контекст для старта — [../promt.md](../promt.md) §8.

> **MVP-срез (apply):** полуавтомат `[Auto][Старт/Стоп]` вместо полного диалога политик.
> Детали UX и реализации — [apply.md](apply.md).

## Проблема

Сейчас запись — ручные тумблеры **Старт/Стоп** (и **Старт/Стоп серии**) на каждом инструменте
(`InstrumentPicker.tsx`). Это требует ручного присмотра и приводит к «нечестному» фону: запись стоит
включённой **ночью / вне сессии**, хотя торгов нет — фон растёт впустую (пинги 7h **не идут** вне
торговых часов — гейт `LivenessProbe`, но намерение в RAM может оставаться включённым). Плюс у инструментов Finam **разные торговые часы и таймзоны** (MOEX, срочка, валюта, US-площадки
с ET + переходом на летнее время) — единого «рабочего времени» нет.

## Идея

Заменить кнопку **Старт/Стоп** на **«Управление записью»** — диалог, где для инструмента (или серии/группы)
задаётся **политика записи**:

- **Ручной** — как сейчас: явные старт/стоп.
- **По расписанию** — автомат: авто-подключение провайдера → запись **только в торговую сессию площадки
  инструмента** (по выбранным дням недели) → авто-остановка на закрытии.
- **Выкл** — не писать.

«Инструмент → авто-connect → запись → авто-stop», с учётом сессии **каждой площадки** (в её таймзоне).

## Зависимости

- **7h** — честная подложка + модель сессии площадки + автомат связи (авто-reconnect / ре-подписка). Без
  автомата связи расписание не сможет надёжно поднимать/держать подключение.
- **Абстракция «торговая сессия площадки»** (общая с 7h-гейтом честности): сессия = атрибут
  exchange/board + **IANA-таймзона** (`Europe/Moscow`, `America/New_York` — US-DST считается сам). Источник
  расписания — см. открытый вопрос ниже; реальный календарь MOEX даёт **7c** (ISS), fallback — эвристика
  `MoexSchedule`.
- **Флаг торгуемости бумаги** от коннектора (`inst.active`, TRANSAQ) — авторитетный realtime-сигнал
  «бумага сейчас торгуется», может заменить/дополнить календарь (проверить на живом, см. 7h).

## История расписаний рынка (`market_schedule`, V013)

Курируемая **версионная история торгового распорядка ПО ДВИЖКУ** (futures/stock/currency) — таблица
`market_schedule` (миграция `V013`). Отдельно от `recording_schedule` (это про запись), но питает ту же
абстракцию «сессия площадки». Двойное назначение:

1. **Форвард-расписание для Supervisor** — вооружение/разоружение записи и warmup считаются от
   версии распорядка, действующей на дату (а не от «текущего» ISS).
2. **Историческое отображение Ганта** — колбаски прошлого рисуются по регламенту, **действовавшему на
   ту дату** (границы 08:50/06:50, вечерняя/ЕТС), а не по сегодняшнему.

Ключевые свойства:

- **SCD-2**: строка = версия, ключ `(engine, effective_from)`; действующая на дату `D` = `max(effective_from) ≤ D`.
- Меняется редко (единицы событий: ДСВД 01.03.2025, ЕТС 23.03.2026, расширение СР 14.07.2026) → строк мало,
  наполняем **вручную/исследовательски**.
- **`confidence`** ∈ `authoritative` (офиц. MOEX: новость/приказ/s1167/ISS) · `empirical` (реконструкция
  по свечам Finam RI/Si/SBER: границы дня + клиринговые перерывы → фазы) · `assumed` (предположение).
  Уточнение = добавление строки с более высокой достоверностью.
- **Роли источников:** дни/аномалии (праздники/переносы/сокращённые + их часы) — из бесплатного ISS
  `dailytable` (есть история с 2018); базовый внутридневной регламент — из `market_schedule` (ISS его
  не версионирует). ISS = **детектор дрейфа** в daily-sync (флаг «проверь», без авто-записи).
- Гранулярность — **по рынку/движку**, не по инструменту (нюанс «валютные опционы вне утренней сессии» —
  follow-up).

Осталось (реализация, по отдельному go): reader `IScheduleHistoryStore` + чтение `/api/sessions` из БД
(вместо «текущего» ISS); daily-sync-детектор; эмпирический бэкфилл 5л по свечам Finam.

### Исключения по дате (`market_schedule_exception`, новое)

Базовое (`market_schedule`) — это *как обычно* выглядит день. Отклонения на конкретную дату (праздник /
сокращённый / сдвиг / внеплановая отмена) — отдельная сущность `market_schedule_exception` (паттерн ISS
`timetable` vs `dailytable`): `date`, `scope` (engine/market/category/instrument), `kind ∈
{no_trade, shifted, shortened}`, границы/фазы, `source`, `confidence`, `resolved`, `note`. **Не
перезаписывает базовое.** Заполняется daily-pre-flight (подтверждение) и вручную. Единый резолвер
(«исключение на дату → иначе базовая версия») обслуживает **и Гант, и Supervisor**.

Полная идеология (два слоя, источники подтверждения, поток pre-flight, решение пользователя) —
**[schedule.md](schedule.md)**.

## Интеграции (внешние сервисы) — подтверждатель расписания

Внешний API ≠ коннектор (request/response + JWT vs stream + Basic). Заводим **отдельную сущность
«внешний сервис»**, которую пользователь создаёт сам — по аналогии с провайдером.

- **Раздел в левом рейле:** новый `NavSectionId = 'integrations'` («Интеграции»), рядом с
  Биржи/Провайдеры. Внутри — список интеграций + кнопка `[+]`.
- **Форма создания** (`[+]`):
  - **Сервис/адаптер** — что́ биндится на код (base URL, auth-flow, endpoints). MVP: `finam`.
    Это аналог `kind` у коннектора; без него система не знает, *как* говорить с сервисом.
  - **Транспорт** — атрибут адаптера: `REST` (MVP) / `gRPC` / `WebSocket` (позже).
  - **Название** — свободный текст (префилл «Finam REST API»), косметика для UI.
  - **Secret** + **Exp date** (advisory: заранее предупреждаем об истечении).
  - После создания → **auth-запрос как health-check** (`tapi_sk → tapi_ak`); ОК → «Интеграция создана».
- **Рабочая область:** кнопка «Запрос» + окно результата (оформить). **MVP — только расписание**
  (`GET /v1/assets/{symbol}/schedule`).
- **Хранение секрета:** пока **на бэке** (переживает рестарт — нужно для авто-pre-flight; в отличие от
  in-memory кред коннектора). Прятать не от кого — пользователь один. После **Keycloak (phase 10)**
  пересмотреть, где и как хранить секреты.

Второй сервис (`Finam gRPC`) и другие — тем же механизмом, позже.

## Связка «запись ↔ базовое расписание ↔ подтверждение»

Supervisor вооружает запись от **резолвера** (базовое + исключения), а не от «текущего» ISS. Перед
арматурой на день — **daily pre-flight** через интеграцию-подтверждатель:

- совпало с базовым → пишем как обычно;
- разошлось → пишем `market_schedule_exception` (Гант + окно записи сразу учитывают) + уведомление
  Warning; пользователь решает «оставить исключение» / «изменить базовое» (новая версия `market_schedule`).
- **Fail-open:** API недоступен → не блокируем, идём по базовому с warning. `no_trade` — осторожно
  (согласие ISS+Finam либо уведомление до снятия с записи).

**Открытый вопрос:** кто указывает подтверждатель для scope — конфиг (per категория/рынок) или хардкод-
дефолты (ISS dailytable = дни/праздники, Finam Schedule = фазы/выходные)? Черновик: хардкод-дефолты +
опц. переопределение позже. Детали — [schedule.md](schedule.md) «Открытые вопросы».

## Модель и решения

- **Политика записи** — на инструмент (с наследованием на серию/группу, как «Старт серии» сегодня):
  `mode ∈ {manual, scheduled, off}`, дни недели, окно (`use_session` | кастомное `from..to`), провайдер
  (`connection_id`), опц. **warmup** (поднять связь за N мин до открытия).
- **Хранение** — таблица `recording_schedule (instrument_id, source_id, connection_id, mode, weekdays,
  window_kind, custom_from, custom_to, warmup_sec, enabled)`. Пока **не** привязано к пользователю
  (user-scope придёт в phase 10 → `user_settings`).
- **Supervisor** — фоновый сервис хоста: тик раз в ~минуту; для каждого `scheduled`-инструмента считает
  «должен ли писать сейчас?» из сессии площадки; **вооружает/разоружает** `RecordingManager` идемпотентно;
  при необходимости поднимает подключение через `ConnectionManager` (автомат связи 7h). Ручные записи
  (`manual`) не трогает — ручной старт всегда перекрывает.
- **Честность фона по построению**: захват идёт только там, где ожидался (сессия) → вне окна фон
  **нейтральный** (не дыра). Снимает саму проблему «фон ночью».

## UI

- Ряд инструмента: вместо **Старт/Стоп** — кнопка **«Управление записью»** (иконка часов/шестерёнки),
  открывает диалог/поповер.
- Диалог: выбор режима (Ручной / По расписанию / Выкл); для расписания — дни недели, «по сессии площадки»
  vs кастомное окно, warmup, провайдер; показ вычисленных **следующего вкл/выкл**.
- Быстрый ручной старт/стоп остаётся доступен в диалоге (override).
- Уровень серии/группы: применение политики на всю серию (аналог «Старт серии»).
- Индикатор эффективного состояния на ряду: `пишет` / `вооружён` (ждёт открытия) / `вне сессии` / `выкл`.

## API / WS

- `GET /api/recording/schedule` (батч по видимым), `PUT /api/recording/schedule` (per instrument / batch).
- Supervisor транслирует эффективное состояние через существующий `recordings$` (+ `connectionStateChanged`
  из 7h); опц. событие `recordingScheduleChanged`.

## Область (MVP)

- `mode` manual/scheduled/off; дни недели + окно «по сессии»; таблица `recording_schedule`.
- Supervisor-тик: авто-arm/disarm по сессии; идемпотентность с ручными.
- Диалог «Управление записью» вместо Старт/Стоп (инструмент + серия).
- US-инструмент пишется по своей ET-сессии (DST-корректно, через IANA-tz).

## Вне области (позже)

- **Warmup/авто-connect с опережением** — если базовый reconnect 7h достаточен, вынести в follow-up.
- **Праздничный/дат-точный календарь** — из **7c** (MOEX ISS); до него — будни + эвристика.
- **User-scope политик** — **phase 10** (`user_settings`, Keycloak).
- **Уведомления об авто-старте/стопе** — **phase 11** (центр уведомлений).
- **Кастомные окна и сложные профили** (пре-/пост-маркет, вечерняя сессия отдельно) — follow-up.

## Критерии приёмки

1. `scheduled`-инструмент **сам стартует** на открытии сессии и **сам стопается** на закрытии; ночью
   запись не идёт, фон не растёт.
2. `manual` = сегодняшнее поведение; ручной старт перекрывает расписание.
3. Инструмент с US-таймзоной пишется по своей сессии с корректным переходом на летнее время.
4. Вне окна записи подложка Ганта **нейтральна** (нет ложных дыр).
5. `tsc --noEmit` + `vitest` + backend-тесты зелёные.

## Открытый вопрос (решить на живом соединении)

Откуда брать сессию: **(1)** статус торгуемости бумаги от TRANSAQ (`active` + tz) — авторитетно, без
календарей; **(2)** эмпирика из `md_trade` (выученные окна) — самокорректируется; **(3)** статический
календарь площадки (нужен для форвард-плана расписания). Вероятный гибрид: (1) как realtime-источник
«сейчас торгуется», (3)/7c как форвард-расписание для вооружения записи заранее, (2) как валидация.
